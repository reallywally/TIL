# chapter3 에이전트 시스템을 위한 UX 디자인

- 에이전트도 화면이 있으니까 UI/UX가 중요하다

## 모달리티

- 사전적 의미로는 특정 목적을 달성하기 위한 ‘방식’, ‘양상’, ‘형태’를 뜻한다.
- 이번장에서는 에이전트가 서비스되는 화면 형태 정도로 이해하자.

### 텍스트 기반 인터페이스

- 클로드 코드, 제미나이 CLI가 대표적인 예시이다.
- 커맨드라인 전문 지식이 있는 사람만 쓰던 터미널을 초보자도 사용하기 편하게 제공한다.
- 핵심 한계는 사용자가 에이전트가 지원하는 능력이나 효과적인 명령 표현 방식을 모르는 경우가 많으며 이를 보완하는 전략이 필요하다.

### 그래픽 인터페이스

- 랭스미스, n8n, 어라이즈, 오토젠 등이 대표적인 예시이다.
- 텍스트 기반 인터페이스 보다 사용이 편하고 직관적이다.
- 단점은 화면의 물리적 공간이 한정적이라 표시할 항목의 우선 순위, 다양한 기기마다 화면 크기, 레이아웃 등 고려해할 사항이 많다.

### 음성 인터페이스

- 아마존 알렉사, 애플 시리 등이 대표적인 예시이다.
- 운전, 요리 기계 조작 등 수동 입력이 어려운 환경에 좋다.
- 사람이 언제나 원하는 바를 정확하게 한번에 말하지 못하기 때문에 매끄럽게 처리하는게 매우 중요하다.
  - 예) "식당 예약을.. 아 잠깐만 음... 내일로 바꿔저

### 비디오 기반 인터페이스

- 아직 상용 서비스는 없지만 얼굴이 보이는 챗봇 정도가 적합해 보인다.
- 인간과 유사항 표정, 제스터를 모사해 소통하여 감정적 뉘앙스를 더할 수 있다.
- 하지만 고품질 비디오 상호작용에는 상당한 연산 능력과 대역폭이 필요하다.

### 모달리티 결합을 통한 매끄러운 경험

- 가장 매력적인 에이전트 UX는 여러 모달리티를 하나의 응집된 사용자 여정으로 결합하는것이다.
  - 예) 운전중에는 음성으로 에이전트와 커뮤니케이션 -> 회의실에 도착하면 텍스트로 대화 -> 비디오 챗봇과 회의 내용 리뷰
- 여러 모달리티가 결합되면 무엇보다 컨텍스트 유지와 지속성이 매우 중요하다.

### 자율성 슬라이더

- UX설계에 종종 간과되지만 매우 중요한 관점중 하나가 에이전트에 부여하는 자율성의 수준이다.
- 클로드 코드, 커서에도 AI가 코드를 만든 후 사람의 승인은 받고 적용하는모드, 자동으로 적용하는 모드, 수정하지 않고 계획만 보여주는 모드가 있다.
- 책에서는 수동, 요청, 완전 자율 3단계 구분한다.
- 자율성 슬라이더 제공은 단순 기능이 아니라 사용자의 전문성과 주체성을 존중하는 메시지를 전달한다.

## 동기식 대 비동기식 에이전트 경험

- 동기식은 GPT와 같이 실시간으로 대화하는것을 의미한다.
- 비동기는 이메일을 자동으로 작성, 전송 후 완료 알림을 보내는것과 같이 시간 간격을 두고 처리되는것을 의미한다,

### 동기식 경험을 위한 디자인 원칙

- 즉시성, 명확성, 반응성이 중요하다.
- GPT, 클로드, 제미나이를 잘 생각해 보면 도움이 될거 같다.

### 비동기식 경험을 위한 디자인 원칙

- 동기식과 반대로 즉시성 보단 유연성, 지속성, 장기적 명료성이 중요하다.
- 작업 상태와 결과를 명확하게 전달해야하는다.
  - 예) 처리 시작 알림, 예상 완료 시간, 결과 보고 등

### 능동적 개입과 침해적 행동 사이의 균형

- 동기식이든 비동기식이든 에이전트가 언제, 어떻게 능동적으로 사용자를 참여시켜야 하는지 결정하는건 어려운 일이다.
  - 예) 이메일을 작성하고 내용 검토를 받을지, 말지
- 단순히 기술적 문제가 아니라 사용자의 워크플로우와 심리 상태에 대한 공감이 필요하다.

## 컨텍스트 유지와 연속성

- 컨텍스트 유지는 기술적 능력이지만 UX적으로 중요하다.
- 효과적인 컨텍스트 유지를 위해 단기와 장기 메모리 모두 적절히 관리해야한다.

### 상호작용 간 상태 유지

- 수천, 수백만 사용자로 확장되면 세션 상태를 메모리에만 두지 말고 DB를 사용하면서 연속성을 보장해야한다.
- 상태관리가 잘 되면 몇 시간을 넘어 며칠이 지나도 사용자가 중단했던 지점에서 이어갈 수 있게 할수 있다.

### 개인화와 적응성

- 코파일럿이 개발자가 적용했던 코드를 분석해서 개발자 취향에 맞는 코드를 만들어 내는것과 같다.

## 에이전트 능력의 기시화

- 사용자에게 에이전트가 무엇을 할 수 있는지 나열하는 데 그치지 않고 효율적으로 활용할 수 있도록 해야한다.

### 신뢰도와 불확실성 커뮤니케이션

- 에이전트는 종종 결정적 규칙이 아닌 통계 모델에 기반한 확률적 환경에서 작동한다.
- 즉, 모든 응답과 결과에 동일한 신뢰도를 가지지 않는다.
- 따라서 불확실성을 효과적으로 전달하고 사용자가 정보에 근거한 결정을 내리도록 돕는데 필수이다.
  - 명시적 진술: '정답일 확률이 90%입니다'
  - 시각적 신호: 아이콘, 색상 기반 경고
  - 행동 조정: 신뢰도가 낮을때 단정적 권고 대신 다른 제안을 제공

### 사용자 지침과 입력 요청

- 아무리 고도화된 에이전트라도 모호하거나 애매한 사용자 입력을 완벽히 해석할 수는 없다.
- 이처럼 모호성에 직면했을때 적절한 질문을 하도록 설계되어야 한다.
  - 예) '일본행 비행기 예약해줘' 라고 했을때 '편도일까요? 왕복일까요?'라 질문
- 한번에 너무 많은 질문은 피해야 한다.
  - 예) '편도일까요? 왕복일까요? 비즈니일까요? 이코노미 일까요? 창가쪽으로 할까요? 비상구쪽을 어떠세요?'

### 우아한 실패(예외처리)

- 에이전트 시스템이 항상 데재로 작동한다는 법은 없다.
- 그래서 실패를 다루는 방식은 성공만큼 중요하다.
- 일반 웹 어플리케이션에서 예외처리를 잘 해야하는것과 같다.

### 상호작용 설계에서의 신뢰

- 신뢰를 얻는건 어려운데 읽기는 쉽다.
- 투명성: 에이전트의 능력과 제약을 명확히 전달
- 예측 가능성: 사용자는 과거 상호작용으로 에이전트의 반응을 예측할 수 있음
- 엣지 케이스도 신중히 처리해야 한다.
