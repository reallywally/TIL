# chapter5 오케스트레이션

- 오케스트레이션이란 단순히 어떤 도구를 언제 호출할지 결정하는것을 넘어 적합한 컨텍스트를 구성해 효과적이고 근거 있는 행동을 유도하는 과정을 포함한다.

## 5.1 에이전트 유형

- 다양한 에이전트 유형을 먼저 이해하자

### 5.1.1 반사형 에이전트(reflex agent)

- 내부 추론 과정을 거치지 않고 '조건이 참이면 도구 실행' 규칙을 따른다.

### 5.1.2 리액트 에이전트(ReAct agent)

- 추론(Reasoning)과 행동(Action)을 반복하여 교차 수행한다.
- 도구를 선택하고 호출한 뒤 결과를 관찰하고 다시 계획을 갱신하는 방식
- 동적 데이터 분석, 문제 해결에 적합

### 5.1.3 계획 후 실행 에이전트(planner-executor agent)

- 계획을 생성하는 **계획 단계**, 각 계획마다 도구 호출을 수행하는 **실행 단계**로 구성된다.
- 추론(계획)에 집중하고 필요한 도구만 호출하여 불필요한 LLM 호출을 줄인다.
- 장점
  - 명확한 분해
  - 디버깅 용이
  - 비용 효율성

### 5.1.4 쿼리 분해 에이전트(Query-decomposition agent)

- 복잡한 질문을 반복적으로 하위 질문으로 나누고 각 하위 질문마다 검색 또는 도구를 호출하여 최종 답변 생성
- 예) 'X와 Y중 누가 더 오래 살았지?' -> 'X의 수명은 얼마였지?', 'Y의 수명은 얼마였지?' -> 종합해서 'X는 10년, Y는 20년 살았으므로 Y가 더 오래 살았다'

### 5.1.5 성찰형 에이전트(Reflection and metareasoning agent)

- 추론과 행동을 넘어 과거 단계를 검토해 실수를 찾아내고 수정
- 리액트 패러다임의 확장판
- 금융 거래, 의료 진단, 중대 사고 대응 등 오류가 연쇄적으로 이어질때 좋음

### 5.1.6 심층 리서치 에이전트(Deep research agent)

- 앞서 설명한 에이전트 유형을 결함한 형태
- 작동 순서
  1. 연구 아젠다 **계획**
  2. 구체적 쿼리 **분해**
  3. API, DB 조회 등 다양한 도구 호출하고 결과의 관련성, 신뢰성 **성찰**
  4. 각 단계에서 LLM 기반 요약, 비판을 활용해 점진적 발전
- 장점
  - 역량: 전문 DB와 복잡한 문제, 단계가 많은 조사를 처리 가능
  - 적응성: 새로운 증거가 나타날때 마다 연구 방향 조정
  - 투명성: 계획과 분해 단계가 명시적으로 드러나 방법론을 감사하고 검토하기 쉬움
- 단점
  - 높은 비용: 비싼 모델, API, 복잡한 연산 및 토큰
  - 지연 시간: 계획, 분해, 성찰 단계가 시간이 오래 거램
  - 취약성: 외부 데이터 소스에 의존

### 5.1.7 에이전트 유형 정리

| 에이전트 유형 | 강점 | 약점 | 최적 사용 사례 |
| --- | --- | --- | --- |
| 반사형 에이전트 | 빠른 응답 | 다단계 추론 불가 | 키워드 라우팅, 단순 조회 |
| 리액트 에이전트 | 유연한 적응력 | 더 높은 지연시관과 비용 | 탐색형 워크플로, 트러블 슈팅 |
| 계획 후 실행 에이전트 | 명확한 작업 분해 | 계획 수립 오버헤드 | 복잡한 다단계 프로세스 |
| 쿼리 분해 에이전트 | 근거가 명확한 검색 정확도 | 다수의 도구 호출 필요 | 리서치, 사실 기반 Q&A |
| 성찰형 에이전트 | 초기 오류 감지 | 추가 연산 및 지연시간 | 고위험, 안전이 중요한 작업 |
| 심층 리서치 에이전트 | 다단계 및 적응형 조사 관리 | 높은 연산 비용과 매우 긴 지연시간 | 장문의 문헌 리뷰 |

## 5.2 도구 선택

### 5.2.1 표준 도구 선택

- 도구와 정의, 설명을 파운데이션 모델에 함께 제공하고 가장 적합한 도구를 선택하도록 모델에 요청한다.
- 효과적인 도구 선택을 위한 가이드
  - 도구에 간결하고 설명적인 이름 붙인다.
    - 예) process_number 대신 calculate_sum
  - 도구의 고유 목적을 한 문장으로 요약한다
    - 예) 두 수의 합을 반환한다.
  - 입력 타입과 범위를 명시한다.
    - 예) x와 y는 0~100 사이의 정수여야 한다.

### 5.2.2 시멘틱 도구 선택

- 모든 사용 가능한 도구를 시맨틱 표현으로 인덱싱하고 시맨틱 검색으로 가장 관련성 높은 도구를 가져 온다.
- 예)

  ```python
  tool_descriptions = {
    "query_wolfram_alpha": "Wolfram Alpha를 이용해 수학식을 계산하거나 정보를 조회한다.",
    "trigger_zapier_webhook": "미리 정의된 자동화 워크플로를 실행하기 위해 zapier 웹훅을 트리거 한다."
    ...
  }
  ```

- 한 번만 처리되면 그 다음부터는 빠르게 조회할 수 있다.

### 5.2.3 계층적 도구 선택

- 도구가 많을 때 유용한 방법이다.
- 쉽게 말해 그룹핑 하는것이다.
- 정확도를 높일 수는 있으나 유지보수가 어렵기 때문에 매우 많은 수의 도구를 사용하는 경우에만 권장한다.
- 예)

  ```python
  tool_groups = {
    "Computation": {
      "description": '''Tools related to mathematical computations and data analysis.''',
      "tools": []
    },
    "Automation": {
      "description": '''Tools that automate workflows and integrate different services.''',
      "tools": []
    },
    "Communication": {
      "tools": []
      "description": "Tools that facilitate communication and messaging.",
    }
  }
  ```

## 5.3 도구 실행

- LLM이 생성한 파라미터로 실제 시스템(API, DB, 함수)을 호출해 작업을 수행한다.
- 왜 이게 중제목으로 분리됬는지 모르겠다.

## 5.4 도구 토폴로지

- 에이전트가 여러 도구(tool)를 어떤 구조로 연결해서 사용하는가를 알아보자

### 5.4.1 단일 도구 실행

- 한번에 하나의 도구만 사용한다
- 예) 도구1 사용 -> 결과 확인하고 필요한 도구 확인 -> 도구2 사용

### 5.4.2 병렬 도구 실행

- 동시에 여러 도구를 사용한다
- 예)

  ```plaintext
          ┌ Search Tool
  Agent ──┤
          └ DB Query
              ↓
          결과 통합
  ```

### 5.4.3 체인

- 일련의 작업을 순차적으로 실행하는 패턴이다.
- 반드시 이전 작업이 성공적으로 완료되어야만 이어서 수행된다.
- 체인 길이가 길어질 수록 오류가 누적될 가능성이 높으므로 최대 길이를 설정하는것이 좋다.
- 대표적 예로 LangChain의 Rimmable이 있다.

### 5.4.4 그래프

- 비선형적 구조로 노드와 노드를 연결하는 형태이다.
- 시간과 비용이 많이 필요하므로 역시 최대 분기 수에 상한을 두는것이 좋다.
- 대표적 예로 LangGraph가 있다.
- **작업이 선형일 경우 체인으로 시작하고 필요한때 그래프로 변경하는것이 좋다.**

## 5.5 컨텍스트 엔지니어링

- **핵심 실천 사항**
  - 컨텍스트 전체 저장이 아니라 유용한 정보만 선택하고 관련성 유지
  - 모델이 예측, 해석 가능한 형태(명료성) 유지
  - 히스토리가 길어지면 핵심 정보만 보존
  - 에이전트의 상태와 작업 단계에 맞게 매 추론마다 프롬프트 컨텍스트를 동적으로 재구성해야 한다.
    - 예)

    ```plaintext
    <1단계>

    목표: 사용자 질문 해결
    컨텍스트:
    - 사용자 질문

    <2단계> (검색 후)

    목표: 검색 결과 분석
    컨텍스트:
    - 사용자 질문
    - 검색 결과
      ```
