# 8. 단일 에이전트에서 멀티 에이전트로

## 8.1 에이전트는 몇 개나 필요할까

- 단순하게 접근하는걸로 시작해서 성능 개선이 필요할때만 추가하는게 좋다.

### 8.1.1 단일 에이전트 시나리오

- 물류 관리용 단일 에이전트로 보면 1개의 에이전트가 재고 확인 툴, 배송 상태 확인 툴 등 여러 툴을 한번에 사용한다.
- 그런데 만약 도구가 16개 까지 확장된다면 1)프롬프트가 복잡해지고 2)에이전트가 올바른 도구 선택에 어려움을 겪어 단일 에이전트의 한계가 발생할 수 있음

### 8.1.2 멀티 에이전트 시나리오

- 여러 에이전트가 공통 목표를 달성하기 위해 협력한다.
- 예제 코드로 보면 도구들의 집합을 잘 분리해서 재고 전문 노드/에이전트, 배송 전문 노드/에이전트 등으로 분리

### 8.1.3 스웜

- 탈중앙화와 자기 조직화를 강조한다.
- 시스템 전체에 대한 관심 없이 자신만의 정책과 규칙을 따른다.
- 대규모 data discovery 작업이나 리서치, 분산 의사결정에 유용하다.

## 8.2 에이전트 추가 원칙

- Task decomposition(작업 분해)
- Specialization(전문화)
- Parsimony(파시모니): 최소한의 에이전트만 추가하도록 권장
- Coordination(조율, 합동, 조화)
- Robustness(견고성)
- Efficiency(효율성)

## 8.3 멀티 에이전트 조율(Multiagent Coordination)

### 8.3.1 민주적 조율

- 에이전트 각자가 동등한 의사결정 권한을 가지고 행동
- 공정한 결과를 얻을 수 있지만 합의에 이르는 과정에서 통신 오버헤드가 발생할 수 있다.

### 8.3.2 관리자 중심 조율

- 중앙집권화하여 하나 이상의 에이전트를 관리자로 지정하여 하위 에이전트의 행동을 감독한다.
- 작업과 책임이 명확해지고 통신 경로가 단순해진다.
- 하지만 관리자 에이전트가 실패하거나 오작동하면 전체 시스템에 장애가 발생할 수 있고 확장성, 적응성이 떨어질 수 있다.

### 8.3.3 계층형 조율

- 에이전트를 여러 수준으로 조직화
- 확장성, 효율성을 좋으나 각 계층 간의 조율이 원활하도록 설게해야해서 복잡성이 높다.

### 8.3.4 액터-크리틱 접근법

- 평가 주도 반복을 가볍게 적용한 형태이다.

## 8.4 애이전틱 시스템의 자동 설계(Automated Design of Agent Systems)

- 스스로를 설계하고 평가하는 작업을 반복하여 개선할 수 있는 시스템이다.
- 내용은 복잡한데 지금 당장 필요한건 아니라서 일단 개념만 익히고 가자

## 8.5 에이전트 통신 기법

### 8.5.1 로컬 통신과 분산 통신

- 소규모에서는 에이전트가 직접 함수 호출, 공유 메모리, 메모리 내 메시지 큐를 많이 사용한다.

### 8.5.2 A2A(Agent-to-Agent) 프로토콜

- 구글이 발표한 프로토콜로 에이전트들 끼리 통신하는 프로토콜이다.
- 에이전트 카드라는 JSON을 이용하며 신원(identity),  기능(capabilities),  엔드포인트(endpoint)로 구성되어저 있다.
- 기본적으로 HTTPS 위에서 작동하는 JSON-RPC2.0을 사용하지만 전송 방식에 구애받지 않도록 설계되었다.
- 아직은 초기단계라서 보안 분야에서 해결해야할 숙제가 많다.

## 8.6 메시지 브로커와 이벤트 버스

- 시스템 규모가 커질수록 점대점(point to point) 통신 방식은 불안정하고 유연성이 떨어진다.
- 이에 대한 일반적인 해결 방안으로 메시지 브로커나 이벤트 버스를 도입한다.
- 아파치 카프카, 레디스, 래빗MQ, NATS 등이 있다.

## 8.7 액터 프레임워크

- 멀티 에이전트 환경에서 상태관리와 순차 처리 보장 등을 제공하는 모델이다.
- 파이썬에서는 레이, 자바에서는 아카가 있다.

## 8.8 오케스트레이션 및 워크플로 엔진

- 템포럴, 에어플로우 같은걸 말한다.

## 8.9 상태와 영속성 관리

- DB, 벡터 스토어, 스토리지 등 저장소를 이용해서 영속성을 관리한다.

## 8.10 마무리

- 단일 에이전트는 소개할게 별로 없어서 그냥 지나가듯이 있었고 주로 멀티 에이전트 환경 관련된 내용이었다.
- 좋은? 내용일 수 있지만 내용이 어려웠고 현재 기준 이상적인 해결 방법이 많았다.
